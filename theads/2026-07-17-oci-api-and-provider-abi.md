## Why this is here

This is the design reasoning behind `cic-module-oracle-cloud`: how the OCI
control plane can be driven from a CIC provider module, and what the module's
ABI should look like. It predates and informs the repo's contract work.

## Key decisions (TL;DR)

- **Language split.** Go for execution (the official OCI Go SDK — no official
  Rust SDK exists); Rust for custody/signing (the Go SDK's `Signer` is
  replaceable, so the private key stays in Rust); Python for schema discovery.
- **Schema extraction.** OCI has no single public OpenAPI catalogue, but the
  Go SDK is a machine-extractable service model: `go/ast` over the generated
  `*_client.go` / `*_request_response.go` / `*_details.go` yields operations,
  HTTP method/path, and request/response fields via struct tags
  (`mandatory`, `json`, `contributesTo:"body|header|path|query"`,
  `presentIn:"body|header"`). A CIC-specific YANG can be generated from this;
  Oracle publishes no YANG for OCI.
- **The module ABI is not `create/update/delete`** but
  `validate / observe / plan / execute / poll / invoke / destroy` — a fixed WIT
  envelope (`cic:provider@0.1.0`) carrying a schema-validated, hashed payload
  (`schema-id`, `schema-version`, `schema-hash`, `encoding`, `data`), so new
  OCI resources never change the WASM ABI.
- **`effective_config` is not a third branch.** It is the read-only, derived
  projection of the state branch used to compare against intent. The model
  settles on **object-level correspondence**, not field/row diff:
  `satisfies(stateObject, intentObject)`, with correlation by CIC `uid`
  (stable) vs `provider_id` (per-realization, changes on recreate).
- **ProofTrace entry points.** The Go SDK `Interceptor` (pre-sign) plus an
  `HTTPRequestDispatcher` wrapper (every physical attempt, retries included);
  `opc-request-id` / `ETag` / `WorkRequest` id become evidence.
- **Versioning risk.** OCI service clients ship breaking changes in *minor*
  versions — the SDK must be pinned exactly and schema-diffed on bump.
- **Custody boundary.** The WASM module carries provider logic, not secrets:
  it gets a `secret_handle` and a narrow host import (sign request, perform a
  permitted HTTP call, read an allowed secret through a handle, emit evidence,
  read monotonic time) — matching the relay's http-executor + vault-adapter.

Az SDK retry-rétege nem egyszerű „próbáld újra háromszor” megoldás.

A Set logikája
A Set lényegében reconciler:

1. Get
2. ha nincs → Create
3. ha van → Diff
4. mutable eltérés → Update
5. speciális eltérés → Action
6. immutable eltérés → Reject vagy Recreate
7. Wait
8. újabb Get
9. Verify
Például:

func (d *VcnDriver) Set(
    ctx context.Context,
    desired DesiredVcn,
) (*ApplyResult, error) {
    current, err := d.findCurrent(ctx, desired)

    if errors.Is(err, ErrNotFound) {
        return d.create(ctx, desired)
    }

    if err != nil {
        return nil, err
    }

    diff := CompareVcn(desired, current)

    if diff.Empty() {
        return &ApplyResult{
            Changed:  false,
            Observed: current,
        }, nil
    }

    if diff.RequiresReplacement() {
        return nil, ErrReplacementRequired
    }

    return d.applyUpdate(ctx, desired, current, diff)
}
Optimista konkurenciakezelés
A Get válaszból kapott ETag átadható az update vagy delete kérés IfMatch mezőjének:

        VcnId: common.String(current.ID),

        UpdateVcnDetails: core.UpdateVcnDetails{
            DisplayName: common.String(desired.DisplayName),

A WebAssembly Component Modelhez tartozó WIT kifejezetten komponensek import/export interfészeinek leírására szolgáló IDL; a működési viselkedést nem definiálja, csak a szerződést. Egy WIT world azt is megadhatja, hogy a komponens mit exportál, illetve milyen hostképességeket importál.

interface provider {
    describe: func() -> module-manifest;

        request: observe-request
    ) -> result<observation, provider-error>;

    plan: func(
        request: plan-request
    ) -> result<execution-plan, provider-error>;

    execute: func(
        request: execute-request
    ) -> result<execution-result, provider-error>;

    poll: func(
        request: poll-request
    ) -> result<execution-result, provider-error>;

    invoke: func(
        request: operation-request
    ) -> result<operation-result, provider-error>;

    destroy: func(
        request: destroy-request
    ) -> result<execution-result, provider-error>;

A kapcsolat nem ez:

intent mező 1 ↔ state mező 1
intent mező 2 ↔ state mező 2
hanem:

IntentObject A  ↔  StateObject A'
A StateObject az intentben deklarált objektum megfigyelt, providerhez kötött realizációja, ezért természetesen több információt tartalmazhat:

OCID-ket;

ETag-et;

lifecycle state-et;

generált értékeket;

provider defaultokat;

időbélyegeket;

kapcsolódó objektumazonosítókat;

bizonyítékokat.

IntentObject
    ↓ realization
StateObject
Nem egyenlőség van, hanem megfelelőség:

StateObject realizes IntentObject
Vagy formálisan:

IntentObject ⊂ StateObject

Az objektumazonosság és a megfelelőség két külön dolog
1. Korreláció
Először meg kell találni, melyik state objektum tartozik az intent objektumhoz:

correlate(intentObject) → stateObject
Elsődlegesen valamilyen CIC-stabil azonosító alapján:

metadata:
  uid: cic-subnet-production
A provider ID nem alkalmas elsődleges CIC-identitásnak, mert újralétrehozáskor változhat:

régi subnet OCID ≠ új subnet OCID
miközben a logikai CIC-objektum ugyanaz marad:

cic-subnet-production
Ezért:

CIC UID
    = logikai objektumidentitás

Provider ID
    = az aktuális realizáció azonosítója
2. Megfelelőség
Miután a pár megvan:

verify(intentObject, stateObject)
A schema vagy adapter objektumszinten értékeli:

a CIDR megfelelő-e;

a logikai route table referencia a megfelelő OCID-re oldódott-e;

a fordított boolean jelentése megfelel-e;

a lifecycle state elfogadható-e;

a provider által generált értékek engedélyezettek-e;

a state tartalmaz-e tiltott vagy idegen konfigurációt.

Így nézne ki a kontraktus
Nem egyszerű mezőtérkép:

display_name: displayName
cidr: cidrBlock
hanem objektumszintű megfelelőségi szabály:

kind: Subnet

correlation:
  intent_identity:
    path: metadata.uid
  state_identity:
    path: metadata.uid

requirements:
  - name: cidr-matches
    assert:
      equals:
        intent: spec.cidr
        state: observed.cidr_block

  - name: route-table-resolves
    assert:
      reference_resolves_to:
        intent: spec.route_table
        state: observed.route_table_id

  - name: public-ip-policy
    assert:
      expression: >
        intent.spec.public_ip_allowed
        ==
        !state.observed.prohibit_public_ip_on_vnic

acceptable_state:
  lifecycle_state:
    - AVAILABLE
A verifikáció kimenete:

verification:
  intent_uid: cic-subnet-production
  state_uid: cic-subnet-production
  correspondence: one-to-one
  result: satisfied

  checks:
    cidr-matches: passed
    route-table-resolves: passed
    public-ip-policy: passed
    lifecycle-state: passed

Mit jelent akkor az „engedélyezett érték”?
Ebben a modellben három külön ellenőrzés van.

Schema-validáció

Realizáció
Létrejött-e hozzá state objektum:

Megfelelőség
A létrejött objektum teljesíti-e az intentet:

satisfies(stateObject, intentObject)
Az általad említett „engedélyezett érték” az elsőhöz tartozik. A state viszont a második és harmadik bizonyítéka.

Tehát a pontos modell
intent branch
    logikai, kívánt objektumok

state branch
    ugyanazon logikai objektumok
    providerrel összekötött realizációi

kapcsolat:
    alapértelmezetten 1 intent objektum
    ↔ 1 state objektum

ellenőrzés:
    objektumszintű, schema által definiált
    szemantikai megfelelőség
Ebben a modellben az effective_config valóban felesleges köztes entitás. Amit korábban annak neveztem, nálad már benne van a state objektum observed/configuration vetületében, de a teljes state dokumentum az azonosítókkal és működési állapottal együtt marad az intent objektum párja.

Az objektumazonosság pontos szabálya
Jól értem, hogy ugyanaz a CIC-azonosító szerepel az intent és state objektumban, miközben a provider ID csak a state-ben van?

intent.uid = state.uid
state.provider_id = aktuális OCI-realizáció
Újralétrehozáskor a uid marad, a provider_id változik.

Egy intent objektum mögött lehet-e több providerobjektum?
Alapesetben 1:1 megfelelés van. De például egy magasabb szintű network intent létrehozhat NAT Gatewayt, route rule-t és NSG-szabályokat. Ezek külön state objektumok lesznek saját intenttel, vagy egy fő objektum belső realizációs részletei?
